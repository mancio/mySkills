# Meteo Multi-Source — Local Web App Builder

## Goal

A locally-runnable Flask app at `http://localhost:5000` where the user types a city and gets a forecast for the next ~30 days. The forecast value shown for every (location, timestamp, variable) cell is the **ensemble average across multiple independent weather providers**, because single-provider forecasts are unreliable. The user can switch between views: **Month grid**, **Day detail**, **Hourly**.

No mock data. Every number must come from a live API call (or its on-disk cache). If a key is missing for a chosen provider, **stop and ask the user** — never invent it.

## Step 1 — MANDATORY interview (always run before any code is written or modified)

Every time this skill is invoked, the agent MUST first ask the user the following questions using the `vscode_askQuestions` tool. Do **not** reuse answers from a previous run; the user may want a different mix today.

Ask all five in a single `vscode_askQuestions` call:

1. **Primary source tier** (single-select):
   - `Free only` (Open-Meteo, no key) — recommended default
   - `Free trial` (Tomorrow.io / Visual Crossing / WeatherAPI / OpenWeather One Call free tier — needs key)
   - `Paid` (Tomorrow.io paid, OpenWeather One Call paid, AccuWeather — needs key + billing)
2. **Free/no-key sources to also include in the ensemble** (multi-select; always at least one). Offer:
   - Open-Meteo ECMWF model
   - Open-Meteo GFS model
   - Open-Meteo ICON model
   - MET Norway (Yr.no) Locationforecast
   - NWS (US-only fallback, free, no key)
   - wttr.in scrape (last-resort)
3. **API keys** (single-select with freeform): `Ask me at build time` vs `I'll paste them now`. For every chosen trial/paid provider, after this answer, individually prompt the user for the key with a freeform question (one provider per question). If the user picks `Free only` and no trial/paid sources, skip this.
4. **Forecast horizon**: confirm 30 days. Warn that most free models only run 7–16 days; beyond the model horizon the app falls back to **ERA5/Open-Meteo climate-normals climatology** for the same calendar day-of-year (averaged 1991–2020). Offer alternatives: `16 days only` or other.
5. **UI runtime**: confirm `Flask + plain HTML/JS at http://localhost:5000` (default) or alternatives (`FastAPI + Jinja`, `Static HTML regenerated by script`).

Persist the answers for **this build only** to `meteo/.build-config.json` (gitignored) so the build steps below can read them. Do NOT read this file before asking — it is written after the interview.

# list of dicts, hourly, in UTC
[{ "ts": "2026-05-08T00:00:00Z", "temp_c": 14.2, "precip_mm": 0.0,

## Ensemble averaging rule (weighted)

For each (timestamp, variable) bucket combine all sources that returned a value for that bucket:

- **Continuous variables** (`temp_c`, `precip_mm`, `wind_kmh`, `humidity_pct`, `cloud_pct`): **weighted arithmetic mean** using `SOURCE_WEIGHTS` from `.build-config.json`. Formula:
  `mean = Σ(value_i × weight_i) / Σ(weight_i)` over the sources that supplied a value for that bucket.
  A source missing from `SOURCE_WEIGHTS` defaults to weight `1.0` (so adding a new adapter never breaks existing builds).
  Also store `min`, `max`, `std`, `n_sources` (computed from the **raw** values, unweighted) so the UI can still show the true spread.
- **Categorical** (`weather_code`, condition icon): mode (most common); ties broken by the source list order chosen by the user. (Weights do not apply to categorical fields.)
- If `n_sources < 2` for a bucket, mark it `"low_confidence": true` in the output.
- Daily aggregates derive from the weighted hourly series: `t_min`, `t_max`, `t_mean`, `precip_total`, `precip_prob = weighted mean`, dominant condition.

Beyond the longest provider horizon, fill remaining days from **climatology** (Open-Meteo Archive API, ERA5, day-of-year mean over the most recent 10 years) and tag those days `"climatology": true` in the API and grey them in the UI.

## Step 1b — Decide source weights at build time

After the user has chosen sources in Step 1 (and provided keys in Step 2), the agent MUST propose a `SOURCE_WEIGHTS` table and confirm it with the user before writing it to `.build-config.json`. Weights are **assigned at build time only for the sources the user selected**, using the rubric below as a starting point.

Default rubric (use these as the proposal; the user may override with freeform input):

| Tier of source | Suggested weight | Examples |
|---|---|---|
| Top-class deterministic NWP / proprietary blend | **1.0** | Open-Meteo ECMWF, AccuWeather, Tomorrow.io paid |
| Strong commercial blend or trial | **0.9** | Visual Crossing, OpenWeather One Call, WeatherAPI |
| Strong regional ensemble service | **0.8** | MET Norway (Yr.no), NWS (when in-region) |
| Global mid-tier model | **0.7** | Open-Meteo GFS |
| Limited-horizon / lower-res model | **0.6** | Open-Meteo ICON (7 d) |
| Scrape / last-resort | **0.3** | wttr.in |

Procedure:
1. Build a proposal dict containing **only the sources the user actually selected**, with the suggested weight from the rubric above.
2. Show the proposal in a `vscode_askQuestions` freeform question, formatted as `key=value` lines, e.g.:
   ```
   open_meteo_ecmwf=1.0
   visual_crossing=0.9
   met_no=0.8
   open_meteo_gfs=0.7
   open_meteo_icon=0.6
   ```
   The question text should say: *"Confirm or edit the per-source weights (higher = more trust). Leave a source out to give it the default 1.0."*
3. Parse the user's reply (one `key=value` per line); fall back to the proposal on parse error.
4. Write the final dict to `.build-config.json` under `source_weights`.
5. Document the chosen weights in the generated `README.md` so the user can later re-tune by editing the JSON and restarting the server.

The weighted mean implementation lives in `src/ensemble.py`; weights are read from `src/config.py:SOURCE_WEIGHTS` at startup.

## Flask routes

- `GET /` → `index.html` with empty state.
- `GET /api/geocode?q={city}` → `[{name, country, lat, lon, tz}, ...]` (from Open-Meteo Geocoding, top 5).
- `GET /api/forecast?lat={lat}&lon={lon}&days=30` → JSON:
  ```json
  {
    "location": {...},
    "sources_used": ["open-meteo-ecmwf", "met-no", ...],
    "hourly": [ {"ts": "...", "temp_c": {"mean":..,"min":..,"max":..,"std":..,"n":..}, ...}, ... ],
    "daily":  [ {"date": "...", "t_min":..,"t_max":..,"precip_mm":..,"condition":"rain","climatology":false}, ... ],
    "generated_at": "..."
  }
  ```
- `GET /healthz` → `{"ok": true, "sources": [...]}`.

## UI requirements

- Single search box at top with type-ahead via `/api/geocode`.
- Three tabs: **Month** (30 cells, mini icon + min/max temp + precip bar; greyed if `climatology`), **Day** (pick a day → hourly chart for temp + precip), **Hourly** (raw 24h table for selected day with per-source breakdown on hover).
- Show `n_sources` badge on every cell; cells with `low_confidence` get a dashed border.
- No external CSS/JS frameworks beyond what's in `requirements.txt` (vanilla JS + small inline SVG charts).

## Build steps (execute in order, after Step 1 interview is complete)

1. Run **Step 1** (interview). Persist answers to `meteo/.build-config.json`.
2. For every trial/paid source the user picked, ask for its API key with a separate freeform `vscode_askQuestions` call. Save to `.env` (key names: `TOMORROW_IO_API_KEY`, `VISUAL_CROSSING_API_KEY`, `WEATHERAPI_KEY`, `OPENWEATHER_API_KEY`, `ACCUWEATHER_API_KEY`). Also ask for a contact email for the MET Norway User-Agent (default `meteo-app/1.0 (you@example.com)`) and store as `MET_NO_USER_AGENT`.
3. Run **Step 1b** (weight assignment): propose default weights for the selected sources, let the user confirm/edit, write `source_weights` into `.build-config.json`.
4. Create `requirements.txt`, `.env.example`, `.gitignore`. Create venv: `python -m venv .venv` then activate `.venv\Scripts\Activate.ps1` (Windows PowerShell) and `pip install -r requirements.txt`.
5. Implement modules in order: `config → cache → geocode → sources/base → each chosen source adapter → climatology → ensemble → views → app`.
6. Each adapter gets a `__main__` smoke test (e.g. `python -m src.sources.open_meteo` prints 3 hourly rows for Rome).
7. Add a CLI smoke test: `python -m src.app --selftest "Rome"` — geocodes, fetches every chosen source, runs ensemble, prints a 5-row daily summary, exits 0.
8. Run the server: `python -m src.app` (Flask development server, host `127.0.0.1`, port `5000`, debug off). Print the URL.
9. Open the page, search for the city the user used in the interview (or `Rome` if none), confirm Month/Day/Hourly tabs render with `n_sources >= 2` for at least the first 7 days.

## Hard rules

- **Always run Step 1.** Never skip the interview, even if `.build-config.json` already exists. The user's preferences may have changed.
- **Never invent API keys.** If the user picks a trial/paid source and refuses to provide a key, drop that source from the ensemble and tell them.
- **Always include at least 2 free no-key sources** in the ensemble (otherwise averaging is meaningless). If the user deselects all of them, re-ask with a warning.
- All HTTP calls: 15 s timeout, 1 retry with exponential backoff, on-disk JSON cache keyed by `(provider, lat, lon, day-bucket)` with 30-minute TTL.
- Respect MET Norway's terms: User-Agent header is mandatory; honor `If-Modified-Since` / `Expires` from response headers.
- Never commit `.env`, `.cache/`, `.build-config.json`, or `.venv/`.
- Climatology-filled days must be visibly tagged in both the JSON API and the UI.
- No telemetry, no external CDN — fully offline-capable once the cache is warm.
